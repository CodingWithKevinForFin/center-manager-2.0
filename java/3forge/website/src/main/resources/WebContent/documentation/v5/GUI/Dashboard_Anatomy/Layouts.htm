Layouts

A dashboard, when saved and loaded can be spread across multiple files for more granular management and concurrent team development of dashboards. These individual files are each called a Layout File. When a Layout file is opened directly, we refer to that as the dashboard. Importantly, a Layout can reference any number of sub-layout files. Potentially, these sub-layouts could also contain sub-layouts, effectively resulting in a dependency tree (See Dashboard > Included Layouts).

 

Layouts & Objects

All objects (Panels, Datamodels, Relationships, etc.)  have an "owning" layout and likewise a layout is a combination of objects. In addition, layouts can also reference other layouts by file name. Each child layout reference must be identified by a unique layout-alias.

 

Scoping

For terminology, when one Layout references another Layout they are referred to as a Parent Layout and Child Layout respectively. A parent Layout's objects can reference objects within the child layout, but the child layout's objects can not reference the parent layout's objects. In other words, the child layout's objects should be considered resources that the parent layout will use.

 

Another way to think of it: any layout at some point can become included in another layout. As such, this layout doesn't have knowledge of what its future parent layouts will contain, hence can't reference their objects, even when included. 

 

Hidden Panels

When a layout contains a panel that is not referenced directly nor indirectly by the root desktop's windows it is considered a hidden panel, meaning there is no way for an end user to access the panel. For example, when a child layout is included into a parent layout, all of its panels would be considered hidden initially. You can unhide a panel by creating a blank panel and choosing Unhide ... from the green settings button.
