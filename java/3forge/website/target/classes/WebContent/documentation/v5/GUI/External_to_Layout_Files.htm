External to Layout Files

Nearly everything is stored within the layout file itself with a few exceptions. These are important to cover, as the layouts will likely have dependencies on them. When moving layouts across instances of AMI, it is important that these are setup consistently across the AMI instances.

 

Datasources

The Definition of a datasource does not live inside the layout, it is stored inside the Ami Center's __DATASOURCE table (see Ami Center Documentation). Layouts simply reference a datasource by name, and it is expected that there is a datasource available matching that name, otherwise the datamodel can not run. This separation makes moving dashboards across environments easier. For example, a datasource may have a particular configuration in the QA environment and a different configuration in Production, but because these details are not stored in the layout, the same layout can easily be used for both. 

 

In circumstances where the datasource must be stored in the layout, one can use an anonymous datasource (see ami.db.anonymous.datasources.enabled property)

 

In the frontend datasources can be accessed through Dashboard > Data Modeler... where they are represented with blue icons as shown in the figure below.

 

 A new Datasource can be added by clicking on the option Attach Datasource. This action will open up the ATTACH DATASOURCE wizard where a new datasource can be attached by user, after specifying it's type and properties.

 

 

 

 

Resources

Larger binary resources like images, audio clips, and video clips are stored and accessed using our resource manager (Menu > Dashboard > Resource Manager). These resources can then be referenced by name inside the layouts. This helps keep layouts small and promotes a separation between the layouts' logic vs. the look & feel.

 

Default StyleSets

Style sets can be saved inside a layout or can be exported and saved in the AMI Server. At that point, the styleset is considered readonly but can be used across layouts. The layouts will reference the style by the style's ID.

 

User Preferences & Settings

As would be expected, these are not saved in the layout and are actually stored on a per user basis.

 

Plugins

Plugins are used to interface AMI with external systems such as SSO/Entitlement Servers, Source Control Management tools, etc. These plugins are configured at the AMI Web (or AMI One) and are then referenced/used by the layout files.

 

Configuration Properties

Like all server applications, AMI has configurable properties used to control how AMI operates. In addition, certain properties can be made available for use as session variables (see Session Variables). These are usually stored in a local.properties file.
